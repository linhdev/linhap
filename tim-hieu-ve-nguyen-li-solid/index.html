Nguyên Lý SOLID Thần Thánh

Trước khi tìm hiểu về design patterns cần phải nắm được nhưng nguyên tắc cơ bản nhất của nó, và SOLID là 5 nguyên tắc cơ bản mà ta cần phải biết với mỗi 1 chữ cái sẽ tương ứng đại diện cho 1 nguyên tắc.

S: Single Responsibility Principle: tức là ý nghĩa của nguyên tắc này là mỗi 1 class chỉ nên đảm nhiệm 1 vai trò nhất định giả sử như là mình có 1 class là Employee trong nó đang có chức năng in ra thông tin của nhân viên mà bây h viết thêm 1 chức năng báo cáo lương của nhân viên vào trong 1 cùng 1 class đó, như thế khi nhìn vào code sẽ rất là phức tạp lúc này nên tách nó riêng là thành 2 class khác nhau: 1 class là Employee 1 class ReportSalary. Mục đích chính của nguyên tắc này giúp giảm đi sự phức tạp của hệ thống. Từ đó sẽ giúp hệ thống khi mà gặp lỗi sẽ dễ dàng phát hiện vấn đề hơn rất nhiều.

O: Open/Closed Principle class should be open for extension but closed for modification. Giả sử mình đang có 1 hệ thống đang hoạt động bây h muốn mở rộng thêm các chức năng thì bắt buộc khi viết code trên hệ thống đó phải giữ nguyên code cũ đang hoạt động, ko được sửa đổi code trước đó của hệ thống vì mình không thể đảm bảo rằng khi sửa đổi chức năng này nó ảnh hưởng tới các chức năng khác hay không, khi implement 1 chức năng mới thì nên extend lại từ chức code trước đó. lúc này sẽ lấy ví dụ

L: Liskov Substitution Principle: nội dung của nguyên tắc thứ 3 này là các object của class con có thể thay thế object của class cha mà ko làm thay đổi tính đúng đắn của chương trình. Thế thì trước khi em nói về nguyên tắc này em sẽ nói qua 1 chút về kế thừa trong java hoặc là trong bất cứ một ngôn ngữ lập trình nào khác chẳng hạn như c sharp. Khi mà sử dụng kế thừa trong java nó sẽ có 1 số nhược điểm sau daayd:

  1. Khi overide 1 method của 1 class cha thì lúc nào cx phải luôn đảm bảo rằng nó tương thích với class cha điều này khiến cả hai bị phụ thuộc vào nhau.

  2. Kế thừa sẽ làm phá vỡ đi tính bao đóng của class cha như mọi người đã biết thì java có 1 tính chất đó là tính bao đóng nó giúp cho việc bảo toàn dữ liệu ko thể sửa xem or sửa đối dữ liệu giả sử khi mà 1 class có thuộc tính là a class b kế thừa class a thì rõ ràng lúc này b có thể sử dụng trưc tiếp thuộc tính của a trong khi đó tính bao đóng chỉ được nhìn thấy thuộc tính trong chính class của nó.

  3. Class con bị phụ thuộc vào class cha: tức là giả sử khi đã code xong 1 chức năng rồi mà bây h mình thay đổi 1 code của class cha sẽ dẫn tới mọi thay đổi của class con cũng bị thay đổi theo và sẽ có nguy cơ dẫn tới nguy cơ gặp lỗi khi đã thay đổi rất là cao.

  4. Kế thừa cũng giúp 1 phần nào đấy tái sử dụng lại code tuy nhiên việc lạm dụng kế thừa sẽ dẫn tới 1 hậu quả đó là tạo ra hệ thống cấp bậc song song.

Thì đó là 1 số nhược điểm của kế thừa tuy nhiên ko thể ko sử dụng kế thừa được vì lợi ích của kế thừa mang lại rất là lớn thì khi sử dụng kế thừa mọi người có thể phải đảm bảo được 1 số nguyên tắc sau đây :

  1. Tham số đầu vào của 1 class con có thể nhận được tất cả các instance của class cha ví dụ như class Animal là cha của class Cat có 1 hàm là feed(Cat c) thf lúc này hàm này chỉ nhận được duy nhất kiểu dữ liệu là Cat có thê đổi lại như sau để chương trình linh động hơn feed(Animal c) thì lúc này hàm feed có thể nhận bât kỳ kiểu dữ liệu nào là instance của Animal

  2. Đối với kiểu trả về của 1 hàm trong class con thì toàn ngược lại so với cái 1 nên trả về kiểu dữ liệu cụ thể giả sử lúc này có 1 class là CatGreen CarBlue extends từ class Cat và hàm buycat lúc này có kiểu trả về trà về là Animal thì lúc này chương tình sẽ không biết là kiểu genreric nào là CatGreen hay CatBlue thì lúc này nên trả về kiểu dữ liệu cụ thể là CatGreen. tuy nhiên điều này sẽ không đúng với 1 số ngôn ngữ lập trình có kiểu dữ liệu là dyamic như javascript, python…

  3. Việc sử dụng exception 1 class con không nên throw ra 1 exception class cha ko có, bình thường thì throws ra 1 exception ở class con mà class cha không có sẽ dần tới ngay lập tức tuy nhiên để giải quyết được điều này có thể sử dụng trycatch trong class con việc này dẫn tới việc ko xác định được exception cụ thể

  4. Giữ nguyên được những gì của class cha đã có ko được thay đổi vì khi thay đổi nó sẽ ảnh hưởng tới toàn bộ hệ thống và rất dễ gây lỗi.

  5. Không được thay đổi value của các trường có access modified trong class cha, tại sao private lại có thể thay đổi được trong khi java quy định là nó chỉ thay đổi được value trong chính nó class đó, thì java có 1 khái niêm là reflection thì hoàn có thể thay đôi được giá tị này khi từ class ngoài truy cập vào.

I: Interface Segregation Princple: nội dung của nguyên tắc này là 1 interface lớn nên tách thành nhiều interface nhỏ.

D: Dependency Inversion Principle: nội dung của nguyên tắc này nói là : class cấp cao không nên phụ thuộc vào class cấp thấp cả hai nên phục thuộc vào abstraction và abstraction không nên phụ thuộc vào detail cả hai nên phục thuộc vào abstraction.
